ThoraxAI

ThoraxAI is a multi-tenant chest X-ray platform for hospitals. It combines React, FastAPI, MongoDB, Redis, PyTorch inference, Grad-CAM explainability, and serverless deployment on Google Cloud. This guide covers the project from first principles through production deployment.

1. Overview

Problem

Hospitals need an internal tool for X-ray triage, staff access, and explainable AI analysis without mixing tenant data.

Solution

A tenant-aware SaaS portal where doctors upload scans, run AI diagnosis, inspect Grad-CAM heatmaps, and read patient-friendly summaries.

Design goal

Quiet operational software, not a marketing site. The interface is dense, predictable, and focused on repeated clinical tasks.

2. What it does

ThoraxAI supports hospital onboarding, invite-code staff enrollment, patient management, scan upload, analysis, and result review. A hospital gets its own tenant context and data store. Doctors only see scans within their hospital.

  • Hospital admins: register a hospital, create invite codes, manage staff, and monitor the tenant.
  • Doctors: manage patients, upload scans, trigger AI analysis, and review reports.
  • Super admins: observe all hospitals across the platform.

Feature list

Database-per-tenant isolation
JWT plus WebAuthn passkey login
OTP email verification
Chest X-ray classification
Grad-CAM overlay generation
Audit log and rate-limit middleware

3. Architecture

Frontend

React 19 SPA hosted on Firebase Hosting. It calls the backend through `VITE_API_BASE_URL` and includes auth token forwarding.

Backend

FastAPI app on Cloud Run. It resolves tenant context, talks to MongoDB and Redis, and runs PyTorch inference for scan analysis.

Data + AI

MongoDB stores app state, Redis handles ephemeral auth and rate-limit state, and model weights are loaded from the backend runtime or Hugging Face.

User -> Firebase Hosting SPA -> FastAPI /api/v1 -> tenant resolver -> tenant MongoDB
                                     -> scan upload -> AI pipeline -> Grad-CAM -> save result

7. Local setup

A local run uses two terminals: one for the backend and one for the frontend. MongoDB and Redis can be started with Docker Compose, or you can point the backend at hosted services.

  1. 1. Install Python 3.11, Node.js, Docker Desktop, and Git.
  2. 2. Create backend and frontend `.env` files from the examples in the repo.
  3. 3. Install backend dependencies with `pip install -r requirements.txt`.
  4. 4. Install frontend dependencies with `npm install`.
  5. 5. Run backend and frontend in separate terminals.

Typical commands

# backend
cd backend
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
uvicorn main:app --reload --port 8000

# frontend
cd frontend
npm install
npm run dev

9. Backend setup

The backend is a FastAPI application with separate route groups for auth, tenants, users, patients, scans, and AI analysis. Startup verifies MongoDB and Redis connectivity and creates indexes before serving traffic.

  • core/settings.py: central configuration and validation.
  • core/events.py: startup/shutdown lifecycle hooks.
  • core/middleware: tenant resolution, rate limiting, audit logging.
  • routes/ai: preprocessing, inference, Grad-CAM, patient summary logic.

Backend boot sequence

1. Load settings and secrets
2. Connect to MongoDB
3. Ping Redis
4. Ensure indexes
5. Start Uvicorn
6. Resolve tenant on each request

10. Frontend setup

The frontend is a Vite SPA. It handles authentication, tenant subdomain forwarding, scan management screens, and result presentation. The API wrapper automatically injects bearer tokens and tenant headers.

  • src/api/client.js: request wrapper, refresh flow, timeout handling.
  • src/api/scans.js: scan and AI endpoint helpers.
  • src/pages/dashboard: scan detail, patient screens, roster, settings.
  • src/context/AuthContext.jsx: user session bootstrap.

Build commands

npm install
npm run dev
npm run build
npm run lint

11. AI pipeline

The AI path is a synchronous request that can be expensive. It preprocesses the X-ray, loads or downloads the body-part-specific checkpoint, runs inference, generates a Grad-CAM overlay, and enriches the result with a patient-friendly explanation.

1. Preprocess

Resize to 224x224, grayscale to 3 channels, normalize.

2. Load model

Load fine-tuned `chest.pth` or download from HF if absent.

3. Predict

DenseNet121 returns class probabilities and confidence.

4. Explain

Grad-CAM highlights the image regions driving the prediction.

5. Persist

Save AI result back to the tenant scan document.

routes/ai/service.py -> preprocessing.py -> inference.py -> gradcam.py -> llm.py -> save_ai_result()

12. Data model

Master database

Houses hospitals, global users, auth records, and audit logs. This database is shared across the platform.

Tenant database

Houses patients and scans for a single hospital. Every request is resolved into one tenant database only.

13. Docker and Compose

The backend container installs PyTorch CPU wheels, builds the FastAPI app, and exposes port 8000. Docker Compose wires together the API, MongoDB, and Redis for local development.

# backend Docker image
cd backend
docker build -t thoraxai-backend:local .

# local stack
docker compose up --build
The backend `.dockerignore` excludes temporary caches, uploads, and local-only runtime output so the image stays predictable. Fine-tuned weights are handled by the production build or runtime download path, not by ad hoc local state.

14. Production deployment

Production uses Google Cloud Run for the backend and Firebase Hosting for the frontend. The backend requires database and cache secrets, plus enough CPU and memory for inference.

  • Backend: Cloud Run container with 2 CPU, 4Gi memory, 300s timeout, concurrency 1.
  • Frontend: Firebase Hosting serves the `dist/` bundle.
  • Secrets: stored in Secret Manager and passed to Cloud Run.
  • Files: scan uploads live in a mounted storage bucket.

Release sequence

1. Build backend image
2. Push to Artifact Registry
3. Apply Terraform
4. Deploy Cloud Run revision
5. Build frontend
6. Deploy Firebase Hosting

15. GCP and Terraform

Terraform manages Cloud Run, Artifact Registry, Secret Manager, Firebase project setup, storage bucket access, and IAM bindings. The live config should be treated as the source of truth for deployment behavior.

Core resources

Cloud Run service, service account, secrets, and storage bucket.

Frontend hosting

Firebase Hosting site and rewrite rules for SPA routing.

Operational note

Do not commit Terraform state or local `.terraform` artifacts.

18. Security notes

  • Tenant isolation: the backend resolves a tenant database before any protected operation.
  • Secrets: production credentials belong in Secret Manager, not in the repo.
  • Uploads: scan files must stay inside the configured upload root.
  • Auth: access tokens and refresh tokens are handled separately and stored in browser local storage for the SPA.
  • Rate limiting: middleware protects public and authenticated paths from bursts.
